# Doc Sync Manager - 文档同步管理器

> 自动识别任务类型，一键生成文档并同步飞书，让开发总结不再繁琐

---

## 1. Skill 简介

**Doc Sync Manager** 是一个基于 SOLO 平台开发的自动化文档管理 Skill，能够自动识别开发任务类型（重构/新需求），根据预设模板一键生成规范的 Markdown 文档，并同步到飞书云文档。

它解决了"代码改完了还要手动写文档、还要手动复制到飞书"的重复劳动问题，适合需要频繁记录开发进度、生成技术文档的开发者使用。

---

## 2. 使用场景

### 你为什么想做它？

作为开发者，每次完成重构或新需求开发后，都要花额外的时间整理文档：

- 手动创建 Markdown 文件
- 复制粘贴到飞书
- 调整格式、补充内容
- 检查文件名是否符合规范

这些机械性的文档工作不仅耗时，还容易遗漏重要信息。

### 你之前遇到了什么麻烦？

- **重复劳动**：每次开发完都要重复同样的文档创建流程
- **格式不统一**：不同人写的文档格式差异大
- **容易遗漏**：测试结果、风险评估等内容经常被忽略
- **同步麻烦**：本地文档和飞书文档需要手动同步

### 做出来之后能省掉哪些动作？

- ✅ 自动识别任务类型，无需手动选择模板
- ✅ 自动生成规范的文件名和标题
- ✅ 一键同步到飞书，返回分享链接
- ✅ 包含完整的文档结构，不会遗漏任何章节

## 3. 创作过程

### 3.1 需求分析

通过 `skill-creator` 技能开始创建，首先明确了：

| 维度       | 内容                           |
| ---------- | ------------------------------ |
| Skill 名称 | doc-sync-manager               |
| 核心功能   | 任务识别 → 文档生成 → 飞书同步 |
| 任务类型   | 重构任务、新需求任务           |
| 依赖工具   | lark-doc 技能、lark-cli        |

### 3.2 文档模板设计

设计了两种标准化的文档模板：

**重构任务文档模板**：

- 基本信息（模块名称、日期、执行人）
- 重构背景（原因和目标）
- 修改内容（涉及文件、关键改动）
- 测试情况（单元测试、集成测试等）
- 风险评估（影响范围、回滚方案）
- 后续优化建议

**新需求任务文档模板**：

- 基本信息（需求名称、开发人员）
- 需求概述（背景、目标、用户价值）
- 实现方案（技术架构、核心逻辑）
- 代码实现（关键代码片段）
- 功能验证（测试用例、验收标准）
- 使用说明、依赖与兼容性

### 3.3 文件命名规范

```
# 本地文件
{日期}_{任务类型}_{模块/页面名称}.md
示例: 2025-05-14_重构_用户中心页面.md

# 飞书文档标题
[{任务类型}] {模块/页面名称} - {日期}
示例: [重构] 用户中心页面 - 2025-05-14
```

### 3.4 工作流实现

```
用户输入 → 关键词识别 → 信息收集 → 文档生成 → 本地保存 → 飞书同步
```

使用 SOLO 的 `lark-doc` 技能调用 `lark-cli docs +create` 命令完成飞书文档创建。

## 4. 使用步骤

### 4.1 触发方式

当用户输入包含以下关键词时自动触发：

**重构任务**：

- "重构完成"、"优化"、"迁移"、"升级"、"重写"

**新需求任务**：

- "新需求"、"开发完成"、"实现"、"新增功能"

### 4.2 操作流程

**Step 1**: 向 SOLO 描述你的任务

```
用户: "用户中心页面重构完成了，帮我生成进度文档同步到飞书"
```

**Step 2**: SOLO 自动识别任务类型

```
→ 识别为重构任务
→ 自动选择重构文档模板
```

**Step 3**: 补充必要信息

SOLO 会询问以下信息：

- 模块/页面名称
- 修改内容概述
- 涉及文件列表
- 关键改动点
- 测试结果
- 其他备注

**Step 4**: 自动生成并同步

```
→ 生成符合规范的 Markdown 文档
→ 保存到本地 /workspace/documents/
→ 同步到飞书云文档
→ 返回飞书文档分享链接
```

### 4.3 使用示例

**示例 1 - 重构任务**：

```
用户: "用户中心页面重构完成了，帮我生成进度文档同步到飞书"
→ 识别为重构任务
→ 收集信息：模块名称、修改内容、涉及文件等
→ 生成: 2025-05-14_重构_用户中心页面.md
→ 同步到飞书: [重构] 用户中心页面 - 2025-05-14
→ 返回分享链接
```

**示例 2 - 新需求任务**：

```
用户: "订单导出功能开发完了，记录一下并上传到飞书"
→ 识别为新需求任务
→ 收集信息：需求名称、实现方案、测试情况等
→ 生成: 2025-05-14_新需求_订单导出功能.md
→ 同步到飞书: [新需求] 订单导出功能 - 2025-05-14
→ 返回分享链接
```

## 7. 总结与思考

### 效率提升

| 指标           | 提升                     |
| -------------- | ------------------------ |
| 文档创建时间   | 从 10-15 分钟 → 1-2 分钟 |
| 文档规范化程度 | 100% 统一，无格式差异    |
| 飞书同步成功率 | 自动化，无需手动操作     |

### 对 AI 工作方式的新感悟

通过这个 Skill 的创作，我深刻体会到：

1. **AI 擅长处理重复性工作**：文档模板生成、格式规范化这类机械性工作交给 AI 最合适
2. **好的 Prompt 就是好的流程**：清晰的工作流程定义 + 结构化的输出要求 = 高质量的 AI 输出
3. **工具联动放大价值**：SOLO + 飞书的联动，让文档管理从"手动复制粘贴"升级为"一键同步"

### 这个 Skill 目前最满意的地方

- ✅ **自动任务识别**：通过关键词匹配自动识别任务类型，减少用户操作
- ✅ **标准化文档模板**：重构和新需求使用不同模板，结构清晰不遗漏
- ✅ **一键飞书同步**：无需手动复制粘贴，直接创建飞书文档

### 后续还想怎么优化

1. 支持更多任务类型（Bug 修复、文档更新、技术调研等）
2. 支持自定义模板，用户可上传自己的模板
3. 添加历史文档查询功能，方便回溯
4. 支持批量文档生成（如 sprint 总结）
5. 集成 AI 自动补充功能，根据代码变更自动填写文档内容
